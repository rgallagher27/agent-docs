Claude Code integration

feature mcp 2026-05-28 Built Dogfood pending

How to wire agent-docs into Claude Code so the LLM can manage your project's docs directly through MCP — create plans, read existing docs, list sections — all without leaving the chat.

Status: the MCP server is built and tested (~12 unit tests, end-to-end smoke against this repo). The full dogfood — Claude Code using agent-docs mcp to write a real doc into this repo — happens in the next session, after you wire the MCP config and restart Claude Code.

Prerequisites

  • agent-docs binary built — from the project root: go build ./cmd/agent-docs.
  • A config file pointing at one or more projects.
  • Claude Code installed and runnable (any version that supports MCP servers).

1. Set up the config

Create ~/.agent-docs/config.toml:

bind = "127.0.0.1:8080"

[[project]]
slug = "agent-docs"
remote = "/Users/USER/Development/agent-docs"
clone_path = "/Users/USER/.agent-docs/clones/agent-docs"
trunk_ref = "main"

The slug appears in URLs (/p/agent-docs/...) and is the value the LLM passes as project to every MCP tool.

For an OSS project, remote can be a git URL (git@github.com:org/repo.git). For local dogfooding, point it at a local path — agent-docs will clone-and-mirror from there.

2. Initial fetch (optional)

Either pre-fetch the bare clone, or skip — the MCP server clones lazily on first project access.

agent-docs fetch --config ~/.agent-docs/config.toml

3. Wire into Claude Code

Two equivalent options.

Option A — CLI (recommended)

claude mcp add agent-docs \
  /Users/USER/Development/agent-docs/agent-docs mcp \
  --config /Users/USER/.agent-docs/config.toml \
  --author-name "Your Name" \
  --author-email "you@example.com"

Option B — manual JSON

Add to ~/.claude.json (user-level) or .mcp.json in your project root:

{
  "mcpServers": {
    "agent-docs": {
      "command": "/Users/USER/Development/agent-docs/agent-docs",
      "args": [
        "mcp",
        "--config", "/Users/USER/.agent-docs/config.toml",
        "--author-name", "Your Name",
        "--author-email", "you@example.com"
      ]
    }
  }
}

Restart Claude Code so it picks up the new server.

What it exposes

ToolRequired argsOptionalEffect
write_doc project, ref, path, content message Commits new or updated HTML to a branch in the bare clone and pushes to origin. Branch must already exist.
read_doc project, path ref Returns the doc's contents at the given ref (defaults to the project's trunk).
list_docs project ref, section Lists entries in a directory at a ref. Empty section lists the root; dir entries get a trailing /.

The tools' schemas (visible to the agent on tools/list) include the section-and-filename conventions (plans/YYYY-MM-DD-{slug}.html, decisions/NNN-{name}.html, etc.) — that's how the LLM knows where to put new docs without further prompting.

End-to-end loop

In Claude Code, say:

List the decisions in my agent-docs project

The agent calls list_docs(project="agent-docs", section="docs-html/decisions") and prints the three ADRs.

Read ADR-001 and summarise its consequences

The agent calls read_doc(project="agent-docs", path="docs-html/decisions/001-repo-canonical-source-of-truth.html"), reads the HTML, and summarises in chat.

Write a new ADR proposing per-project ACLs. Use today's date and the next free ADR number.

The agent calls list_docs to find the next free number, then write_doc(project="agent-docs", ref="main", path="docs-html/decisions/004-per-project-acls.html", content="...full inline-CSS HTML..."). The commit lands in the bare clone and pushes to your remote — your local working tree picks it up on the next git fetch.

Browse via the web UI

Run the HTTP server alongside the MCP server (different process, same binary, same config):

agent-docs serve --config ~/.agent-docs/config.toml

Then open:

  • http://localhost:8080/p/agent-docs/docs-html/index.html — trunk alias for any committed doc
  • http://localhost:8080/p/agent-docs/main/docs-html/decisions/ — auto-generated section listing
  • http://localhost:8080/p/agent-docs/feat-x/docs-html/plans/2026-06-01-foo.html — preview a doc on a feature branch (the URL you'd paste in Slack)

Troubleshooting

SymptomLikely cause & fix
Agent says the tools aren't available The MCP subprocess didn't start. Check Claude Code's MCP logs for the agent-docs stderr stream — common causes are a bad binary path, a missing config file, or invalid TOML.
write_doc errors with "resolve branch X: reference not found" Target branch doesn't exist in the bare clone. Push it from your working tree first, or run agent-docs fetch to mirror new branches.
write_doc errors with "push ...: ..." The bare clone can't push to origin — usually credentials. Make sure the remote is reachable from the host running agent-docs (SSH key, deploy key, GitHub App token, etc.).
Doc URL 404s in the web UI If the doc was written via MCP, it's there — try http://localhost:8080/p/{slug}/main/{path}. If it was written outside agent-docs (e.g. by you on your laptop), run agent-docs fetch to pull the new commits into the bare clone.
Auto-generated index instead of a committed one Expected when no index.html is committed at that path. Commit one if you want a hand-authored index; otherwise the auto-generated listing is the intended fallback.

Solo vs team mode

Same binary either way. Solo: run on your laptop, point Claude Code at it via stdio. Team: run agent-docs serve on a shared host (the same process need not handle MCP — that runs in each dev's Claude Code session pointing at the team config). Multi-user auth and per-project ACLs are not yet implemented (Q6 / Q8 in the architecture's open questions).

Other harnesses

agent-docs speaks vanilla MCP over stdio. Any MCP-compatible client should work — OpenAI Codex, OpenCode, Cursor, Zed, and Continue all have MCP support. Configuration mechanics differ but the wire protocol is the same. If you wire it up to a non-Claude-Code harness, file an issue and add to this page.
