ADR-003 — Branch-aware URLs with preview lifecycle

adr 003 2026-05-28 Accepted

Every URL is branch-scoped. Trunk is the alias. Branches have an explicit lifecycle so spike URLs survive merge and deletion — drop one in Slack, it still resolves a week later.

Status: Accepted, 2026-05-28. Resolves Q11. Operational details (Q12 branch discovery, Q13 snapshot TTL, Q14 local-mode flags) are deferred to MVP implementation.

Context

Given ADR-001 (docs are in git) and ADR-002 (HTML is what's stored), the next question is how the web UI surfaces docs across git branches.

Work happens in feature branches; the web has a single URL space. Two scenarios drive the design:

  • Branch-stable work — a dev on feat/foo writes docs as part of the change; they merge into trunk alongside the code.
  • Spike previews — a dev wants a shareable URL for comment before (or instead of) merging. The URL must be stable enough to drop in Slack and re-open a week later, even after the branch is gone.

A docs system that only renders trunk is hostile to how teams actually work. Pre-merge review of an LLM-authored design exploration is exactly the use case agent-docs exists to support.

Decision

Every URL is branch-scoped. Trunk is just the alias. Branches have an explicit lifecycle.

URL shape

  • Canonical: /p/{project}/{ref}/path/to/doc.html.
  • {ref} is a branch name, tag, or short commit SHA.
  • The trunk branch (default main) is also reachable via the clean form /p/{project}/path/to/doc.html.
  • The SHA form /p/{project}/{sha}/... is the permalink — it always resolves as long as the commit is reachable in the bare clone, even after the branch is deleted.
  • Branch names with slashes (feat/foo) are path-segmented. A reserved sentinel refs/heads/ disambiguates only when a branch name would clash with a real doc path.

Branch lifecycle

EventURL behavior
Branch pushed (new)Server indexes the ref; URLs go live within seconds.
Branch updatedOutput refreshes; URL unchanged.
Merged into trunkBranch URL keeps working but renders a "merged into main at abc1234" banner. Doc-level URLs alias forward to the post-merge location on trunk.
Deleted unmergedServer keeps a snapshot of the last SHA. URL renders a "branch deleted" banner. The permalink /p/proj/{sha}/... still works.
Snapshot retentionConfigurable per project. Default: indefinite on hosted; per-project TTL on self-hosted.

Consequences

Wins

  • Spike URLs are first-class — drop one in Slack, it still works after merge or deletion.
  • Matches the GitHub mental model (/blob/<branch>/<path>) — familiar to every dev.
  • PR-preview pattern emerges naturally; agent-docs can later integrate with GitHub / GitLab to post preview links on PRs.
  • Permalink form means a doc URL pinned at a commit SHA survives anything that doesn't garbage-collect the commit itself.
  • Branch deletion isn't catastrophic — the doc URL transitions to "archived" state rather than 404'ing.

Costs

  • Server must track branch lifecycle events. Webhooks (GitHub / GitLab) are the obvious mechanism; polling is the fallback for self-hosted git remotes that don't push events. Q12 open
  • Need a GC policy for snapshots eventually — indefinite retention isn't free on hosted instances. Q13 open
  • URL scheme has edge cases when branch names look like path segments — handled by the refs/heads/ sentinel.
  • Cross-branch references in docs default to "same branch as the referrer". Explicit cross-branch refs need an explicit {ref} in the link's href. (Authors writing branch-stable docs can keep using relative links naturally.)
  • Local mode (one tree, no {ref} segment) needs slightly different routing than shared mode — the binary has both code paths. Q14 open

Status & revision

Accepted 2026-05-28. Operational details (webhook setup, snapshot retention, local-mode flags) are deferred to MVP implementation and will produce follow-up ADRs as those decisions land.
