ADR-001 — Repo-canonical source of truth

adr 001 2026-05-28 Accepted

Project docs live in docs-html/ in each project's git repository. The agent-docs server holds bare clones per project. SQLite holds derived index only — it can be rebuilt from the git tree at any time.

Status: Accepted, 2026-05-28. Resolves Q1 from Architecture › Open questions. Foundational for ADR-002 and ADR-003.

Context

agent-docs needs a place to put the docs. The choice of where they "really" live shapes everything downstream: how the server is structured, what happens when the server is down, whether docs version with code, how PR review works, how multi-user concurrency is handled, and whether we can support self-hosting without lock-in.

Constraints that drove the decision:

  • Docs should be reviewable in PRs alongside the code they describe.
  • A server outage must not lose work — a hard requirement for a tool teams will rely on.
  • Branching and merging must work end-to-end: work happens in feature branches.
  • Multi-user OSS positioning means no proprietary storage layer that locks teams to one vendor.
  • Operationally cheap to self-host: ideally a single binary plus a directory.

Four options considered (full tradeoff table in Architecture › Source of truth):

OptionOne-line summary
A. Repo-canonical, server mirrorsDocs in docs-html/ in each project repo; server holds working copies.
B. Server-canonical, no repoDocs in the server's DB; nothing in the repo.
C. HybridServer-canonical with snapshot export to the repo.
D. Repo-canonical, local-only serverSame as A but no shared instance.

Decision

Option A — repo-canonical, server as a git-backed layer.

  • Each project's docs live in a directory (default docs-html/, configurable per project) at the root of that project's git repository.
  • The agent-docs server holds bare clones per project. It does not maintain working checkouts.
  • Reads serve doc content directly from the requested ref in the bare clone.
  • Writes via MCP use go-git's in-memory tree manipulation to produce a commit on the target ref and push to the project's remote.
  • SQLite holds derived state only — project list, ref metadata, parsed-from-HTML doc metadata, full-text search, users, ACLs. It is rebuildable from the git tree at any time, by design.

Consequences

Wins

  • Docs version with code. PR review of doc changes is free.
  • Server outage doesn't lose any doc — every dev's local clone has them.
  • Branching, merging, conflict resolution are git's job, not ours.
  • Self-hosting is "run the binary, point at a directory" — no proprietary store to back up.
  • Cross-project search and a unified UI are still possible via the bare clones; SQLite just indexes them.

Costs

  • Write latency includes a git commit + push round-trip (typically < 1 s; can be parallelised for batched writes).
  • The server needs git access to every registered project remote — deploy keys, GitHub App tokens, etc. Q6 open
  • Branch discovery requires either webhooks or polling. Q12 open
  • Concurrent writes to the same branch need a short branch-level lock around the commit-push window. Q8 open
  • Schema changes that touch all docs (renaming a section, moving paths) require migration commits across every project — there is no central rename.

Status & revision

Accepted 2026-05-28. Foundational. ADR-002 (storage format) and ADR-003 (URL model) both build on this decision.
