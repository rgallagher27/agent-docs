ADR-002 — HTML-canonical, no source language

adr 002 2026-05-28 Accepted

The committed file is fully self-contained HTML with inline CSS, authored by the LLM following the style template. The server stores and serves it verbatim — no render step for doc content. Metadata is derived from semantic HTML on ingest, not declared in frontmatter.

Status: Accepted, 2026-05-28. Resolves Q2, Q3, Q15. Revises an earlier (rejected) "Markdown source + HTML render" direction that briefly appeared in architecture drafts.

Context

Given ADR-001 (docs live in git), what format is the committed file?

The instinctive answer — and the one earlier architecture drafts assumed — was a Markdown source with YAML frontmatter, rendered to HTML by the server. This is the Hugo / Jekyll / Astro model and would have been familiar.

The problem: the LLM does not produce Markdown-shaped output. A typical agent-docs page contains:

  • Inline SVG diagrams (see the system sketch in architecture.html).
  • Custom grid layouts (.grid-2, card decks).
  • In-page mockups, visual comparisons, side-by-side option tables with rich cell content.
  • Colored panels with semantic meaning (.panel-good, .panel-warn).
  • Status badges, tag chips, checklist-style trackers.

Markdown can't express any of that without HTML escape hatches. Treating Markdown as the source means the LLM either writes degraded content or embeds raw HTML constantly — at which point "source" is no longer Markdown in any useful sense. The abstraction is lossy.

Decision

HTML is canonical. There is no source language.

  • The committed file in docs-html/ is fully self-contained HTML with inline CSS, authored by the LLM following docs-html/_assets/STYLE_AND_TEMPLATE.md.
  • The server stores and serves the file verbatim — no render step for doc content.
  • Server-side rendering is limited to chrome: search UI, ref selector, "merged" / "deleted" banners, and section-index pages that are auto-generated from directory listings if not committed.
  • Metadata (title, status, tags, doc type, typed cross-doc links) is derived from the doc's semantic HTML on ingest — <h1>, .badge classes, .tag chips, data-link-type attributes — and indexed in SQLite. It is not declared in a separate frontmatter block.
  • The LLM may invoke a scaffold_doc(...) MCP helper that returns a boilerplate-wrapped scaffold (CSS, breadcrumb, header, footer) for the LLM to fill in. This is a UX convenience, not a storage-format change — the resulting committed file is still full inline HTML.

Consequences

Wins

  • The LLM keeps full HTML expressive range — SVG, mockups, custom layouts, anything HTML allows.
  • No source-to-output divergence. The committed file is the doc.
  • Files open via file:// with no infrastructure. Self-contained means self-contained.
  • Matches the existing working convention exactly — no migration friction.
  • The style template ships with the product and is meaningful to LLMs by itself. They can produce conforming docs without a custom renderer in the loop.

Costs

  • Inline CSS repeats across every file (mitigated by scaffold_doc; rarely a real issue in practice).
  • Site-wide style updates require rewriting every doc — a rare event, but real. Tooling will be needed for fleet-wide template upgrades eventually.
  • Parsing HTML for metadata is more brittle than reading YAML frontmatter. The parser must be conservative — when in doubt, omit, never guess.
  • The style template (STYLE_AND_TEMPLATE.md) becomes load-bearing — it is the spec the LLM follows. Changes to it ripple across all future docs.
  • "Programmatic mutation" tools (e.g. an update_plan_step helper) become HTML-edit helpers, not structured-field updates. The LLM is expected to do the editing; MCP enforces conventions and commits.

Status & revision

Accepted 2026-05-28. Builds on ADR-001. ADR-003 builds on this (specifically: there is no render step to worry about across refs).

Native doc — no markdown source.
