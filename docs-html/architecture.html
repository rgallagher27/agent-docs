System sketch

agent-docs is a single Go binary that exposes three surfaces, all reading from the same on-disk doc tree:

Project repos are canonical. The server holds bare clones per project, exposes the MCP surface that agents write through (commits land on a target ref), and serves docs (HTML, as authored) plus navigation chrome (search, ref selector, section indexes) on demand from any ref. SQLite holds the cross-cutting index — projects, refs, users, sessions, search — but never doc content.

Tradeoff: source of truth

Where do docs really live? This shapes everything downstream.

OptionWhere docs liveProsCons
A. Repo-canonical, server mirrors In docs-html/ in each project's git repo. Server keeps working checkouts. Docs version with code. PR-reviewable. Branch/merge semantics free. Server is rebuildable from git clone. No data loss if server dies. Write path needs git ops (commit, push, pull). Concurrent writes need handling. Cross-project search needs an index.
B. Server-canonical, repo-free In the server's filesystem / DB. No docs-html/ in repos. Simpler write model. Real-time collab possible. Docs decoupled from code state. Checkout an old branch, docs don't follow. Server outage = no docs anywhere. No PR review.
C. Hybrid (server-canonical, snapshot export) Server is primary; periodic export commits a snapshot back to the repo. Best UX, still get a git-tracked snapshot for audit. Two sources of truth most of the time. Reconciliation is the hard part of this design and there's no good answer.
D. Repo-canonical, local-only server Same as A but server only runs on the dev's machine, no shared instance. Trivially simple. Zero infra. Loses cross-project view, team visibility, shared search. Becomes a glorified static-site generator.
Pick: A. Repo-canonical, server mirrors via git. The server is a read-mostly aggregator and a write-front that translates typed MCP calls into git commits on a working checkout. This preserves the property that makes the existing convention valuable (docs live with code) and gives us the multi-user surface we need for free, via git itself.

Tradeoff: where the MCP server runs

Given (A), there are still two viable shapes for where mutation happens:

OptionWhere MCP runsProsCons
1. Local-only Each dev runs agent-docs on their laptop. MCP mutates files in their local checkout. They commit and push. Zero infra. Each write is naturally serial. No conflicts. No team-shared instance unless every dev runs one and aggregates somehow. Cross-project view fragmented.
2. Both (local or shared) The same binary serves either mode. Solo devs run it locally; teams run a shared instance. MCP clients point at whichever. Same product, two deployment shapes. Solo path is the team path without auth. Familiar OSS positioning. Shared instance has to handle concurrent writes (different devs' agents writing simultaneously). Mitigated by per-doc advisory locks + git's own conflict resolution.
3. Shared-only Always a remote server. Local agents call it. Single canonical instance simplifies the mental model. Worse solo UX. Needs network for everything. Higher barrier to first-run.
Pick: 2. The binary serves both. agent-docs serve with no auth flags runs in single-user mode against a local checkout. agent-docs serve --auth on a VPS or in Docker runs in team mode against a configured set of repos. Same code path; the difference is whether SQLite/users is populated and whether the bind address is loopback.

Tradeoff: doc storage format

The LLM's output is full of inline SVG diagrams, in-page mockups, custom grid layouts, colored panels — things a Markdown source language can't express. The doc is HTML; trying to derive it from a more constrained source is a lossy abstraction. (Earlier drafts of this section recommended a Markdown-source + HTML-render model — that was wrong; it gave up authoring expressiveness for tooling friendliness that didn't actually matter.)

OptionStorageProsCons
i. HTML-canonical (current convention) The committed file is fully self-contained HTML with inline CSS, written by the LLM following the style template. Server stores and serves as-is. LLM keeps full expressive range — SVG, mockups, custom layouts. Files open via file://. No build step, nothing to "keep in sync". Matches the existing working convention. Storage and authoring are the same model. CSS repeats in every file. Updating site-wide style means rewriting every file. LLM has to retype boilerplate per doc (mitigable via a scaffold_doc MCP helper).
ii. HTML body + server-templated chrome LLM writes only the body content; server templates inline CSS + breadcrumb + footer at write time. Committed file is the full self-contained HTML. LLM types less per doc. Site-wide style updates are easier in principle. Site-wide style changes produce huge git churn (every doc re-rendered, every ref). Couples docs to the server's templating. Loses portability — if you take docs out of agent-docs, the chrome leaves with them.
iii. Markdown source + HTML render Canonical is .md with YAML frontmatter; HTML is rendered on demand. Tooling-friendly. Familiar SSG model. Easier programmatic mutation in principle. Markdown can't express the LLM's full output. SVG, custom panels, mockups, side-by-sides all become escape hatches. Rejected — this was the earlier wrong direction.
iv. Structured-only data, rendered to HTML Stored as YAML/JSON; server renders to HTML. Strongly typed. Even more constraining than markdown. Throws away the LLM's HTML-native authoring entirely.
Pick: i. The committed file is HTML, full stop. The LLM authors it directly following the style template; the server stores and serves it as-is. This matches the existing working convention and preserves the LLM's expressive range. If CSS-repetition toil becomes annoying later, the server can offer a scaffold_doc MCP helper that returns a boilerplate-wrapped scaffold for the LLM to fill in — that's a UX tweak, not a storage-model change.
Implication for "structured" features: things like ADR auto-numbering, status, step trackers, typed cross-doc links don't need a structured source format. They're already expressible via semantic HTML in the style template (.badge classes, .checklist, data-* attributes on <a> for typed links). The server parses the HTML on ingest to populate the search/list index in SQLite — metadata is derived from the doc, not declared alongside it.

Tradeoff: branches & previews

Work happens in feature branches; the web UI has one URL space. Two scenarios drive the design:

  • Branch-stable work — a dev on feat/foo writes docs as part of the change; they merge into trunk alongside the code.
  • Spike previews — a dev wants a shareable URL for comment before (or instead of) merging. The URL must be stable enough to drop in Slack and re-open a week later.

Both must be first-class. A docs system that only renders trunk is hostile to how teams actually work.

OptionModelProsCons
A. Main only Server only renders the trunk branch. Branches are invisible until merged. Trivially simple. Spike review is impossible. Defers all collaboration to PR diffs. Misses the whole point of "share a URL".
B. Branch-in-URL Every URL is branch-scoped: /p/{proj}/{ref}/path. Trunk aliases to the clean form. Mirrors GitHub. Explicit. Any pushed branch is browsable. Need explicit semantics for what happens after a branch merges or is deleted.
C. Branch-in-URL + preview lifecycle Same as B, plus explicit lifecycle: merged branches alias forward to their post-merge location; deleted-unmerged branches keep an archived snapshot at the last SHA. Spike URLs stay valid. Branch URLs survive their branch. The Vercel-style preview pattern works natively. Comment threads in Slack don't 404 after a merge. More moving parts. Needs a GC policy for snapshots eventually.
D. Server-side drafts (not git branches) The server has its own "draft" concept, separate from git. Decouples from git's branch model. Reinvents branches. Reconciliation problem. Splits the source of truth — which is the thing we just decided not to do.
Pick: C. Every URL is branch-scoped; the trunk URL is just the alias. Branches are first-class objects with a lifecycle. Spike previews are the natural shape, not a special case.

URL shape

Canonical: /p/{project}/{ref}/path/to/doc.html.

  • {ref} is a branch name, tag, or short SHA.
  • The configured trunk (default main) gets an alias: /p/{project}/path/to/doc.html serves the same content as /p/{project}/main/path/to/doc.html.
  • Branch names with slashes (feat/foo) are path-segmented. A reserved sentinel (refs/heads/) disambiguates only when a branch name would clash with a doc path.
  • A short SHA URL (/p/proj/a1b2c3d/...) is the "permalink" form — survives branch deletion forever.

Branch lifecycle & URL behavior

EventURL behavior
Branch pushed (new)Server indexes the ref; URLs go live within seconds.
Branch updated (more commits)Rendered output refreshes; URL unchanged.
Branch merged into trunkBranch URLs stay valid but render a banner ("merged into main at abc1234"). Doc-level URLs alias forward to the post-merge location on trunk.
Branch deleted, unmergedServer keeps a snapshot of the last SHA. URL renders with a "branch deleted" banner. The permalink (/p/proj/{sha}/...) still works.
Snapshot ages outConfigurable retention. Default: keep indefinitely on hosted; per-project TTL for self-hosted with disk constraints.

Two paths for the spike-and-share use case

PathHow it worksWhen it fits
Shared / hosted instance Dev pushes the spike branch. Server picks it up via webhook (or poll), renders, serves at /p/proj/spike-name/.... Dev pastes the URL in Slack. Team has a shared instance; pushing the branch is acceptable. The common case.
Local + reverse tunnel Dev runs agent-docs serve locally; working tree is the truth. The server is exposed to teammates by whatever tunnel / VPN the team uses — Tailscale, Cloudflare Tunnel, ngrok, plain LAN, anything. agent-docs has no preference; it's just an HTTP server. Spike isn't pushable yet (half-broken, contains secrets, pre-branch). Team has any way to reach the dev's machine. Async review while the dev's machine is up.
In local mode, the server's default ref is "whatever the working tree is on" — git checkout other-branch switches what the server renders, no restart. Explicit {ref} in the URL reads from the local clone's bare-repo cache if the branch exists there.

Storage implication

The earlier sketch hinted at per-project working checkouts on the server. For multi-branch support those become bare clones. Reads pull the doc's HTML blob from the target ref and serve it directly; writes use go-git's in-memory tree manipulation to produce a commit on the target ref and push. There's no render step — the HTML committed to git is what gets served. Section indexes and chrome (search bar, ref selector, "merged" banner) are still generated server-side, but the doc content itself is delivered as authored.

Data model

Two stores, with a hard line between them:

On disk (git-backed)

The truth. Per-project docs-html/ directories — full self-contained HTML files, exactly as the LLM produced them.

  • plans/YYYY-MM-DD-slug.html
  • features/{name}.html
  • decisions/NNN-{name}.html
  • reviews/YYYY-MM-DD-slug.html
  • index.html, PROJECT_STATUS.html, etc.

SQLite (index + identity)

Rebuildable from on-disk content. Never canonical.

  • projects — id, slug, git remote, bare-clone path, trunk ref
  • refs — project, ref name, kind (branch/tag/snapshot), state (active/merged/deleted), last_sha, last_seen_at, retention
  • docs — path, project, ref, section, title, status, updated_at, parsed metadata (json — tags, badges, link refs extracted from the HTML on ingest)
  • links — from_doc, to_doc, kind, ref-scope
  • fts — FTS5 over rendered text
  • users, tokens, acls
  • audit — who did what, when, which commit

Invariant: deleting the SQLite file and re-running agent-docs reindex reproduces a working instance. The git tree is the only thing that matters.

Deployment shapes

Solo / local tier 0

agent-docs serve on the dev's machine. Points at a directory of project checkouts. Loopback only, no auth. Single user.

Team / self-hosted tier 1

Docker image / docker-compose. Persistent volume for git checkouts + SQLite. OIDC or API tokens. Behind a reverse proxy on the team's network.

Public / hosted tier 2

Multi-tenant managed instance (community-run or commercial). Tenant isolation, billing if applicable. OSS binary, same image.

Auth & access control

  • MCP clients authenticate with API tokens scoped to a user and a set of projects. Tokens are issued via the web UI; agents store them like any other secret.
  • Web users authenticate via OIDC (Google / GitHub / generic) or local password for self-hosted. Sessions in a signed cookie.
  • ACLs are per-project: read, write, admin. ADR-worthy enough to deserve its own doc once we settle on it; deliberately simple for now.
  • Single-user mode skips all of this — no users table populated, no auth middleware, loopback bind. Toggle on by passing nothing.
