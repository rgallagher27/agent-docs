Vision

vision v0 2026-05-28

Why agent-docs, who it's for, and what it deliberately isn't. This is the north star; specifics live in architecture.html.

The problem

AI coding agents produce enormous amounts of useful intermediate work — implementation plans, design tradeoffs, decision rationale, code-review notes — and most of it evaporates as soon as the chat window scrolls. The good stuff doesn't survive to the next session, let alone to a teammate or to future-you trying to remember why a thing was built the way it was.

Three concrete failure modes today:

  • Lossy outputs. Plans live in transient chat or scratch files. ADR-worthy decisions get made and forgotten. Reviews scroll off the screen.
  • No shared convention. Every team (and every harness) reinvents the doc layout. Claude Code's plans don't compose with Codex's plans don't compose with whatever a teammate did in OpenCode last Tuesday.
  • Generic doc tools are agent-blind. Notion, Confluence, Outline are fine page-stores but have no typed concept of "this is a plan tracker, mark step 3 done" or "this is ADR-014, here are the consequences". An LLM writing into them is back to free-form prose.

Meanwhile, the pattern that does work — dense, dark, single-file HTML docs sitting in docs-html/ next to the code, with status badges and tradeoff tables and ADR templates — is just a private convention. It needs an LLM to know about it on the way in (via a system prompt) and a human to maintain it on the way out (via memory and discipline). Both are leaky.

What agent-docs is

A small, self-hostable service that bakes the working convention into a product the LLM harnesses can talk to natively.

  • An MCP server. Coding agents (Claude Code, OpenAI Codex, OpenCode, Cursor, Zed, etc.) call typed tools — create_plan, update_plan_step, create_decision, list_open_questions — that mutate the canonical doc set for a project.
  • A doc model with primitives, not pages. Plans, features, ADRs, and reviews are first-class entities with known shapes. The MCP surface is typed against them; the renderer knows how to display them.
  • A navigable web UI. The dense, dark, no-JS aesthetic the convention already uses. Cross-project search, status dashboards, ADR timelines. Drill into any project; drill into any doc.
  • Git-backed. Docs live in the project repo (in a docs-html/ or .agent-docs/ directory) and travel with the code. The server is a layer over those repos, not a replacement for them. If the server disappears tomorrow, every doc is still on every dev's laptop.
  • Single Go binary, OSS, self-hostable. One person on a laptop, a team on a private network (behind any tunnel or VPN), or a community-run hosted instance — same binary, same model.

Who it's for

AudienceWhat they get
Solo devs using AI heavily The trail of work is captured properly. Plans don't go missing between sessions. ADRs accumulate without ceremony.
Small teams Shared visibility into what the AI is doing across everyone's branches. A single place to review pending plans and recent decisions.
OSS maintainers Contributors and reviewers see plan trackers and ADRs as first-class artifacts in the repo, with a hosted view for the project site.

What makes it different

vs.Difference
Notion / Confluence / Outline LLM-first interface, not retrofitted "AI features". Typed doc primitives. Lives in your repo, not a SaaS silo.
Plain markdown in docs/ Enforced shape (so all plans look like plans), web UI, cross-project view, MCP surface so every harness writes the same way.
Static site generators (Hugo, Astro, etc.) Designed for the read+write loop with an agent, not just the read side. Mutation is typed, not file-editing.
Agent-specific memory / context tools Output is durable, human-reviewable HTML — not internal scratch state. Shared across harnesses, not locked to one.

Non-goals

  • General-purpose wiki. If you want a free-form page tree with rich embedding, use Outline or Notion.
  • Realtime collaborative editing. One writer per doc at a time is fine; the typical writer is an agent, and conflicts are resolved by git.
  • Task tracker. Tasks belong in Linear / Jira / GitHub Issues. agent-docs covers the docs about work, not the work items themselves.
  • Replacement for the existing in-repo convention. agent-docs is that convention, productised. It should feel like the natural next step, not a fork.

Where this is going

Order of operations, roughly:

  1. Lock the architecture (source of truth, MCP runtime location, doc storage format). See architecture.html.
  2. Walking-skeleton MVP: single-binary Go server, one project, three doc types (plan, decision, review), wired to one LLM harness (Claude Code) via MCP. Web UI reads from disk.
  3. Multi-project, multi-user. Auth, ACLs, cross-project search.
  4. Second harness integration (Codex or OpenCode). Validate the convention holds across LLMs.
  5. Hosted reference instance. Public docs site for agent-docs itself, hosted on agent-docs. (Dogfood all the way down.)
Tone check: this is a tool for engineers who already use AI heavily and have opinions about how their work should be recorded. It should feel opinionated and tight, not flexible-and-empty. The right comparison points are gh, direnv, graphite — not enterprise wikis.
