Walking-skeleton MVP

plan mvp 2026-05-28 In progress

Prove the end-to-end loop. One project, one branch, one MCP client (Claude Code), one HTML doc created via MCP, pushed to git, served at a branch-scoped URL, edited, merged, URL still resolves.

6 / 9 steps complete
Auto-index landed phase
ADRs 001-003 locked
Single user mode for MVP

Scope

In scope

  • Single Go binary agent-docs that serves both HTTP and an MCP server (stdio).
  • One project registered via a config file.
  • Bare-clone storage per project (in ~/.agent-docs/clones/).
  • Three MCP tools: write_doc, read_doc, list_docs.
  • HTTP routes: /p/{proj}/{ref}/... serving HTML from the bare clone; /p/{proj}/... aliasing to trunk.
  • Section-index auto-generation (when {section}/index.html isn't committed).
  • Claude Code MCP wiring (config snippet + README instructions).
  • End-to-end validation: create doc → push → fetch via URL → edit → merge → trunk URL serves merged content.

Out of scope (for MVP)

  • Authentication / multi-user. Single-user, loopback only.
  • Cross-project web UI / dashboard.
  • Search.
  • Branch lifecycle banners ("merged", "deleted").
  • Forward-aliasing of merged branch URLs.
  • Snapshot retention / GC.
  • agent-docs render CLI.
  • Second MCP client (Codex / OpenCode).
  • Webhook ingestion — manual agent-docs fetch trigger for now.

The bar is "Claude Code writes a doc, it lands in git, I can open it at a branch URL". Everything else waits.

Approach

Thin vertical slice through the full stack, then widen. Each step crosses every layer so the loop is end-to-end as early as possible — preferring "works for one case" over "complete for the typical case" at every choice.

Package layout:

cmd/agent-docs/
  main.go              entry point; subcommands: serve, fetch, version
internal/
  config/              load config.toml; projects list
  gitstore/            bare-clone wrapper around go-git
  server/              HTTP handlers; route /p/{proj}/{ref}/...
  mcp/                 stdio MCP server; write_doc / read_doc / list_docs
  render/              section-index generator (no doc rendering)
go.mod, go.sum

Validation criteria

MVP is "done" when, from a clean machine:

  1. Install agent-docs, run agent-docs serve.
  2. Configure a project pointing at a test git remote.
  3. From Claude Code, invoke write_doc on branch feat/test-doc to create plans/2026-05-29-hello.html.
  4. The file appears in the project repo on branch feat/test-doc, committed and pushed.
  5. Visit http://localhost:8080/p/{proj}/feat/test-doc/plans/2026-05-29-hello.html — the doc loads.
  6. Visit http://localhost:8080/p/{proj}/feat/test-doc/plans/ — auto-generated section index lists the doc.
  7. From Claude Code, invoke write_doc again with updated content — the URL now serves the new content.
  8. Merge feat/test-doc into main. http://localhost:8080/p/{proj}/plans/2026-05-29-hello.html serves the doc on trunk.

Risks & unknowns

RiskMitigation
MCP Go SDK churn or limited tool surface. Be ready to hand-roll over stdio JSON-RPC. The MCP wire protocol is small enough.
go-git's commit + push performance on a bare clone. Acceptable for MVP; revisit if > 1 s per write. Could fall back to shelling out to git for hot paths if needed.
Path / branch-name disambiguation in URL parsing. Use the refs/heads/ sentinel from ADR-003 when needed. Keep the MVP project's branch names simple (no slashes) to defer.
Claude Code MCP config format may shift between releases. Document for the current format; revisit if it changes. The README section is small to update.
Auth deferred — but the binary is bound to loopback. If a dev exposes it to a network accidentally, anyone reachable can write. Bind to 127.0.0.1 only in MVP. Refuse to start with a non-loopback bind address unless an explicit --unsafe-no-auth flag is set. ADR for the auth model lands before any multi-user work.

Decisions made during implementation

StepDecision
Step 1 Module path is github.com/rgallagher/agent-docs for now. Rename later via go mod edit -module … + a tree-wide find/replace of imports if the canonical repo home turns out different.
Step 1 Stdlib-only — no cobra, no urfave/cli. flag.NewFlagSet per subcommand handles the three commands we need. Re-evaluate only if subcommand count balloons.
Step 1 Stub subcommands print their parsed flags to stdout then return a wrapped errNotImplemented referencing the tracker step. Exit code 1; tracker step number in the error message. Keeps the binary honest about its state.
Step 1 run() extracted from main() and takes (args, stdout, stderr) so it's table-test-driveable without touching os.Args or process I/O.
Step 2 TOML over JSON, via github.com/BurntSushi/toml v1.6.0. First (and so far only) external dep. JSON would have been stdlib-only but hand-edited config readability matters too much for self-hosters.
Step 2 (*Config).Validate() mutates to apply defaults and validates. Single entry point. Idempotent. Avoids the two-phase trap where someone forgets to call ApplyDefaults before Validate.
Step 2 Config-or-die: serve and fetch require a valid config to proceed. No silent zero-project runs.
Step 2 --addr on serve overrides the config's bind. CLI flag wins over file value — standard ergonomics.
Step 2 Default bind is 127.0.0.1:8080 (loopback), enforcing the safety stance from the MVP plan's risks table. Future --unsafe-no-auth flag will gate non-loopback binds explicitly.
Step 3 Picked go-git/v5 (pure Go, no shell-outs). Adds ~10 transitive deps; acceptable.
Step 3 Bare-clone mirror semantics: Fetch overrides go-git's default refspec with +refs/heads/*:refs/heads/* + +refs/tags/*:refs/tags/*. Otherwise incoming refs land in refs/remotes/origin/* and refs/heads stays stale — broke TestFetch_PicksUpRemoteChanges initially. Mirror semantics is exactly what a docs-store bare clone wants.
Step 3 Commit is a single op (write blob → build tree → encode commit → update local ref → push). Callers don't deal with the local/remote split. If push fails after the local ref moves, the bare clone briefly leads remote; next fetch reconciles. Accept the risk for MVP.
Step 3 In-memory tree rebuild: buildTree walks parent subtrees, creates missing intermediate dirs, persists each modified tree as a new object. Stays inside go-git's object store APIs — no shelling out to git.
Step 3 RefKind enum (RefBranch / RefTag) rather than a bool — extensible for future ref types (notes, snapshots).
Step 3 Tests shell out to the system git binary for fixture setup (bare init, commit, push). Pure-go-git fixture-building would mean re-implementing what we're testing. The system git dependency is acceptable — tests run on a dev machine or CI, both have it.
Step 4 Threaded context.Context through run() → cmdServe / cmdFetch. main() sets up a signal.NotifyContext for SIGINT/SIGTERM. Tests pass context.Background() or a cancellable ctx as needed. Cleaner than handling signals inside cmdServe directly.
Step 4 Go 1.22+ ServeMux patterns ({rest...} wildcard). No third-party router. The branch-vs-path ambiguity is resolved in a single handler that consults gitstore.ListRefs — if the first segment is a known ref, treat as explicit; otherwise trunk-relative.
Step 4 Open now does a post-clone mirror Fetch to populate refs/heads/* for non-default branches. Without this, TestHandler_ExplicitRefBranch failed: go-git's default clone refspec puts non-default branches in refs/remotes/origin/* only, so ListRefs didn't see feat-x. Same root cause as the step 3 Fetch fix — now centralised in Open.
Step 4 Project-root URL /p/{slug} (no trailing slash) → 301 to /p/{slug}/. Keeps relative links in served docs (e.g. plans/foo.html's ../index.html) resolving against the project root, not against the server root.
Step 4 SHA-permalink form (/p/{proj}/{sha}/... per ADR-003) deferred. isKnownRef only matches against ListRefs output (branches + tags); SHA-prefixed URLs currently fall through to trunk-relative interpretation, which 404s. Follow-up task; not blocking MVP.
Step 4 Cache-Control: no-cache on doc responses for MVP. Branch-stable docs can take a longer cache later (keyed by commit SHA, immutable per commit); not worth the cache-invalidation complexity now.
Step 5 internal/render as its own package — pure functions, no IO, no git imports. Server composes IndexPage structs from request context and calls RenderIndex. Clean dependency direction (server → render; never reverse).
Step 5 html/template for the generator (not raw string concat) — pulls in automatic context-aware escaping for free. TestRenderIndex_EscapesUserContent verifies a <script> in H1 and an "><img> in an entry href both get neutralised.
Step 5 Fallback layered after the ReadBlob attempt, not before. A committed index.html always wins over auto-generation. Matches Apache / nginx DirectoryIndex semantics.
Step 5 No-trailing-slash directory URLs (/p/proj/main/plans) get a 301 to /p/proj/main/plans/. Required for relative links in the generated index to resolve correctly against the listed dir. Mirrors how every web server handles directory URLs.
Step 5 The generated index's inline CSS lives in a const indexCSS beside the template, not duplicated per file. Mirrors the style template's palette and typography so generated pages look the same family as LLM-authored docs.
Step 5 convertEntries skips an index.html in the listing — if the dir has one committed, we'd have served it; if it doesn't, advertising a non-existent link would mislead.
Next concrete action: step 6 — MCP server. stdio transport, three tools: write_doc(project, ref, path, content), read_doc(project, ref, path), list_docs(project, ref, section?). Writes go through gitstore.Commit. Reads through gitstore.ReadBlob / ListDir. Decide on Go SDK vs hand-rolled JSON-RPC on first commit (mcp-go is the obvious candidate — check API stability before depending on it).
